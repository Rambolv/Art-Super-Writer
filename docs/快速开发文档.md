# ASR Writer 快速开发文档

> 最后更新: 2026-07-09
> 本文档帮助理解项目的整体数据流：从导入文档 → AI分析 → AI写作的全过程。

---

## 一、项目文件夹结构

每个项目创建时自动生成以下文件夹：

```
项目名/
├── chapters/              ← 存放所有章节文件 (.md)
├── drafts/                ← 草稿版本
├── reviews/               ← 审查记录
├── reviewHistory/         ← 审查历史
├── worldTreeData/         ← 剧情结构数据（主干剧情、支线、伏笔等）
├── characterProfiles/     ← 角色档案（性格、行为模式、关系等）
├── characterAnalysis/     ← 角色深度分析
├── readingReports/        ← 阅读节奏报告（吸引力、情感浓度等）
├── config/                ← 项目配置
├── projectLog/            ← 操作日志和分析报告
├── fuzzyMemory/           ← 上下文压缩（旧章节摘要，节省AI容量）
├── skills/                ← 写作技巧库
│   ├── external/          ←   外部引入的技巧
│   └── self/              ←   本项目的优劣总结
├── styleMemory/           ← 风格记忆
└── referenceNotes/        ← 参考笔记
```

**导入时**：上传的 `.md`/`.txt` 文件直接保存到 `chapters/` 文件夹。

## 二、点击「分析」后发生了什么

点击分析按钮 → 调用 `analyzeProject()` → 执行 8 步多轮对话 → 结果存入各文件夹。

### 流程总览

```
第1步：探测能力 → 问LLM"你的上下文上限是多少？"
第2步：分批阅读 → 把全文分批发给LLM，LLM回复"已读"
第3步：阅读体验 → LLM评价整篇的吸引力和情感浓度
第4步：故事结构 → LLM分析主线/支线/伏笔/世界观
第5步：角色印象 → LLM逐一描述所有角色的性格和行为模式
第6步：整体感受 → LLM讲节奏判断/最想知道的事/对下一章的期待
第7步：技能评价 → LLM以编辑视角评价优缺点
第8步：输出JSON → LLM输出结构化数据，前端解析保存
```

### 各步骤详情

#### 第1步：探测模型能力

**发给LLM**：
```
请简要回答：你的上下文 Token 上限是多少？
建议每次输入的 Token 数在什么范围最优？
回答格式如："上限100K，最优30K-50K"。
```
参数：`max_tokens=200, temperature=0.1`

**前端处理**：从回复中提取数字 → 计算每批大小（预留指令和输出空间）。

---

#### 第2步：分批阅读

**发给LLM的 系统提示词**（仅第一轮）：
```
你是一个耐心的小说读者。我会分批发送小说内容给你阅读，
你读完每批后只需回复"已读"即可。
等我发出分析指令时，再基于所有已读内容做分析。
```

**每批发给LLM**：
```
请阅读以下第{i+1}/{total}部分内容，读完后回复"已读"：

[分批的章节内容]
```
参数：`max_tokens=1024, temperature=0.1`

**前端处理**：
- 如果在最优容量内（如30K-50K），整篇发送
- 超出最优范围则按段落边界分割成多个批次
- 超出模型总容量（>80%）则分成多部分分别阅读，不分析章节

---

#### 第3步：阅读体验

**发给LLM**：
```
你已读完了整部小说的全部内容。现在请以一位普通读者的身份，
给出阅读体验。注意：全文已作为单章处理，请针对整篇内容回答：
阅读吸引力(1-10)、情感浓度(1-10)、作用标签(引入/铺垫/冲突/转折/高潮/收尾/过渡/揭秘)。
用自然语言回答，不必JSON。
```
参数：`max_tokens=4096, temperature=0.7`

**保存**：→ `projectLog/分析报告_第1轮_阅读体验.md`

---

#### 第4步：故事结构

**发给LLM**：
```
接下来分析故事结构：
1. 主线脉络，按顺序列出关键事件节点，标注重要性和先后顺序。
2. 有哪些正在发展的支线，当前进展、与主线关联度(1-10)。
3. 作者埋了哪些还没揭晓的伏笔，按读者最想知道的迫切程度排序。
4. 世界观有什么独特设定？
用自然语言回答。
```
参数：`max_tokens=4096, temperature=0.7`

**保存**：→ `projectLog/分析报告_第2轮_故事结构.md`

---

#### 第5步：角色印象

**发给LLM**：
```
请识别出故事中所有出现过的角色，一个都不要漏。
对每个角色标注ta是主角、反派、配角还是主要角色。
然后逐一给出：
- 一句话评价
- 最大特点（举例）
- 遇到困难时的应对方式
- 做决定时的风格
- 倾向稳赚还是冒险
- 最在意什么和最怕什么
- 和谁亲近和谁冲突
- 言行是否一致（有没有出人意料的行为，是否合理）
- 如果要让ta做违背本性的事需要什么前提
用自然语言回答。
```
参数：`max_tokens=8192, temperature=0.7`

**保存**：→ `projectLog/分析报告_第3轮_角色印象.md`

---

#### 第6步：整体感受

**发给LLM**：
```
现在给出整体感受：
1. 目前节奏感觉如何（太平/有点慢/正好/偏快）？
2. 读完你最想知道的三件事，按迫切程度排序。
3. 如果下一章让你来写，你希望看到什么？
4. 有什么想提醒作者注意的？
用自然语言回答。
```
参数：`max_tokens=2048, temperature=0.7`

**保存**：→ `projectLog/分析报告_第4轮_整体感受.md`

---

#### 第7步：写作技能评价

**发给LLM**：
```
最后，请以资深编辑的视角评价这部小说：
1. 本文写得好的地方（优点，举例说明）。
2. 本文的不足之处（缺点，举例说明并提出改进建议）。
3. 给作者的综合建议（1-2条最核心的）。
用自然语言回答。
```
参数：`max_tokens=4096, temperature=0.7`

**保存**：→ `projectLog/分析报告_第5轮_写作技能评价.md`

---

#### 第8步：输出 JSON

**发给LLM**：
```
根据以上所有分析，请输出以下JSON格式的分析结果（只输出JSON，不要其他文字）：

{
  "plot_structure": {
    "trunk": [{"chapter":1,"event":"事件描述","importance":8}],
    "branches": [{"name":"支线名","start_chapter":1,"status":"发展中|接近收尾","relation":7}],
    "buds": [{"chapter":1,"foreshadow":"伏笔描述","urgency":"高|中|低"}],
    "roots": [{"chapter":1,"category":"地点|规则|历史|其他","content":"设定描述"}]
  },
  "characters": {
    "角色名": {
      "role":"主角|反派|配角|主要角色",
      "tag":"一句话评价",
      "traits":["特点1","特点2"],
      "decision_style":"做决定的方式",
      "risk_attitude":"面对得失的态度",
      "cares_about":["在意的事1","在意的事2"],
      "fears":["恐惧的事1","恐惧的事2"],
      "bottom_line":"底线",
      "relationships":{"角色A":"关系描述"},
      "trust_pattern":"信任倾向",
      "conflict_style":"冲突处理方式",
      "consistency_issues":["一致性警告"],
      "breaking_point":"突破底线的条件",
      "behavioral_model":"行为模式总结"
    }
  },
  "rhythm": {
    "per_chapter":[{"chapter":1,"page_turner":6,"emotion":4,"role":"作用标签"}],
    "overall_pacing":"整体节奏",
    "top_questions":["最想知道的事1","最想知道的事2"],
    "next_chapter_hope":"下一章期待"
  },
  "memo": {
    "writer_advice":["写作建议1"],
    "warnings":["警告1"]
  },
  "skills": {
    "strengths":[{"aspect":"优点方面","examples":"例子","detail":"详细说明"}],
    "weaknesses":[{"aspect":"不足方面","examples":"例子","detail":"详细说明","improvement":"改进建议"}],
    "overall_advice":"综合建议"
  }
}
```
参数：`max_tokens=65536, temperature=0.3`

### 所有操作都实时写入磁盘

系统中的每个操作都会**立即**写入对应的磁盘文件：

| 操作 | 实时写入 | 写入位置 |
|------|:--------:|---------|
| 创建项目 | ✅ await 等待完成 | 创建完整文件夹结构 + `project.json` |
| 导入文件 | ✅ | `chapters/原始文件.md` |
| 分析第1~5轮报告 | ✅ 每轮结束立即写 | `projectLog/分析报告_第X轮_*.md` |
| 分析完成输出JSON | ✅ | `projectLog/分析报告_第5步_综合JSON.json` |
| 剧情结构解析 | ✅ | `worldTreeData/world_tree.json` |
| 角色数据解析 | ✅ | `characterProfiles/characters.json` |
| 节奏数据解析 | ✅ | `tensionReports/rhythm.json` |
| 写作台保存章节 | ✅ | `chapters/chapter_XXX.md` |

**注意**：分批阅读时的文本分割仅在内存中进行，不会在磁盘上生成 `full_text.md` 或 `part_XXX.md` 文件。原始导入的 `.md` 文件是唯一的源文件。分析完成后生成 `chapter_001.md` 作为统一的章节文件。

页面展示时直接从磁盘文件读取：

```
分析完成 → 实时写入各磁盘文件
                          ↓
页面展示 → 调用服务器API读取磁盘文件 → 显示
                          ↓
页面刷新 → 重新从磁盘读取（无需 localStorage）
```

## 三、AI 写作时从本地提取的信息

在写作台点击「✍️ 开始写作」后，`writeChapter()` 函数从以下来源提取信息，拼成提示词发给LLM：

### 提取的数据来源

写作时不是直接从磁盘文件读取的，而是从**内存**中读取。内存中的数据是页面启动时从磁盘文件加载好的。

| 信息类别 | 实际读取来源（内存） | 最初来自磁盘文件 |
|---------|-------------------|----------------|
| **剧情进展** | `p.worldTree` | `worldTreeData/world_tree.json` |
| **角色行为模型** | `p.characters` | `characterProfiles/characters.json` |
| **阅读节奏数据** | `p.rhythm` | `tensionReports/rhythm.json` |
| **写作备忘录** | `p.writerMemo` | `projectLog/分析报告_第5步_综合JSON.json` |
| **上一章结尾** | `p.chapters` | `chapters/chapter_XXX.md` |
| **写作方向** | 用户在表单填写 | — |
| **行为规律参考** | 代码内置常量 | — |

### 拼装后发给LLM的提示词结构

```
系统提示词：
  你是一位擅长[小说类型]小说创作的作家。
  [行为规律参考 - 约束角色行为一致性的原则]
  写作注意事项（5条规则）

用户提示词：
  请写第X章。

  上一章（第X章）结尾：
  [上一章最后200字]

  【当前故事进展】
  主线已推进到以下节点：
  · 1. 第1章：事件描述（重要性8）
  · 2. 第3章：事件描述（重要性9）
  读者最想揭晓的伏笔：
  · 第2章埋下：伏笔描述

  【角色行为模型——写作时必须严格遵守】
  [角色名] 一句话评价
  行为模式：做决定方式；面对得失态度；信任倾向；冲突处理；底线；突破底线的条件
  在意的：xxx
  恐惧的：xxx
  ⚠️ 需注意的一致性：xxx

  【当前阅读节奏】
  整体节奏：正好
  读者最想知道：1、xxx 2、xxx
  读者对下一章的期待：xxx

  【写作建议】
  · xxx

  【⚠️ 警告】
  · ⚠️ xxx

  【写作方向】
  [用户填写的写作方向]

  请写出一章约[字数]字的小说内容。直接输出章节正文。
```

## 四、核心设计原则

1. **LLM 以读者身份分析** — 不是文学专家，是刚读完小说的普通读者
2. **不强制心理学框架** — 不用 OCEAN/DarkTriad，读者自然描述即可
3. **分析结果回流写作** — 剧情结构/角色池/节奏数据在写作时注入AI提示词
4. **一次写好** — 不依赖多次审查修正
5. **展示和注入并行** — 页面展示给用户看，同时数据喂给AI写作
6. **LLM 自己归类输出** — 前端只保存数据，不写解析逻辑
7. **全文作为单章处理** — 不分章节分析，避免AI误判章节边界

## 五、文件夹名称对照（代码内部 vs 用户理解）

| 代码内部文件夹名 | 用户理解名称 | 说明 |
|-----------------|-------------|------|
| `chapters/` | 章节文件 | 存放所有 .md 章节 |
| `worldTreeData/` | 剧情结构数据 | 主线、支线、伏笔、世界观设定 |
| `characterProfiles/` | 角色档案 | 每个角色的性格和行为模式 |
| `readingReports/` | 阅读节奏报告 | 吸引力、情感浓度、读者想知道的事 |
| `fuzzyMemory/` | 上下文压缩 | AI容量不够时，旧章节自动压缩为摘要 |
| `projectLog/` | 操作日志 | 所有分析报告和分析过程记录 |
| `skills/` | 写作技巧库 | 本文的优缺点总结和外部写作规则 |
