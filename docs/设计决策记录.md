## 2026-07-09 修复重启后分析结果不显示
- **问题**: 重新打开软件后，前端无法看到之前的AI分析结果
- **根本原因1**: `initApp()` 只对 `DB.currentProject`（项目列表第一个）调用 `syncProjectFromDisk`，但世界树/角色池/写作台等页面使用 `getLockedProject()` 获取数据。如果锁定项目≠当前项目，锁定项目的数据不会被加载
- **根本原因2**: `lockProject()` 和 `selectProject()` 没有调用 `syncProjectFromDisk` 加载磁盘数据
- **根本原因3**: `selectProject()` 有 `!p.analyzed` 守卫条件，已分析项目不会触发磁盘同步
- **改动**:
  1. `initApp()`: 同时同步锁定项目和当前项目的磁盘数据
  2. `lockProject()`: 锁定项目时同步磁盘数据
  3. `selectProject()`: 移除 `!p.analyzed` 守卫，总是同步磁盘数据
  4. 项目页文件夹统计: 修复使用错误属性
- **数据流**: 项目元数据（含 analysisCount）→ 服务器 project.json → 前端 initApp 创建项目对象 → syncProjectFromDisk 用 analysisCount 查找 v{N} 子文件夹 → 读取分析数据到内存


## 2026-07-10 story_state.json 实现
- **决策**: 实现 story_state.json 系统，增强连续性引擎数据完整性
- **数据结构**:

## 2026-07-13 LLM调用架构重构
- **问题**: `callLLM()` 和 `callLLMStream()` 是完全独立的 LLM 调用实现，重复了 `LLMSession` 类的逻辑。不支持 Slot 模式，导致本地LLM分析每次调用都重新 prefill，7+ 次累积 60-70s
- **修复**: 
  - 新增 `_callViaSession()` 统一入口，callLLM/callLLMStream 改为薄封装
  - 无论 Slot 还是 Stateless 模式，都走 LLMSession
  - 清除残留的旧代码（-88行）
- **效果**: Slot 模式下后续调用只发新增消息，KV cache 只处理增量，本地LLM分析预计从 ~60s 降至 ~15-20s

## 核心设计原则（已确认不可变更）

1. **LLM 以读者身份分析** — 不是文学专家，是刚读完小说的普通读者
2. **OCEAN/DarkTriad 不作为强制框架** — 读者自然描述即可
3. **分析结果回流注入 Writer** — 世界树/角色池/节奏数据在写作时注入提示词
4. **写作尽量一次写好** — 不依赖多次审查修正
5. **展示和注入并行** — 页面展示给用户看，同时数据喂给 Writer
6. **LLM 自行归类输出** — 不是前端解析 JSON 字段，是 LLM 自己输出结构化数据

## 世界树与模糊记忆的关系

- **世界树**: 精确的剧情结构数据库，不压缩不模糊，为剧情精准服务
- **模糊记忆**: 解决 AI 上下文窗口限制的降级方案，旧章节层级压缩（L0→L4）
- **两者关系**: 各司其职，并行服务于写作流程。模糊记忆读取世界树的精确数据注入 Writer，但不对世界树数据做模糊化处理

```
├── config/                ← 设置：配置
├── projectLog/            ← 历史日志：日志条目
├── fuzzyMemory/           ← 模糊记忆：上下文+报告
├── skills/                ← 写作技巧库：规则
├── styleMemory/           ← 写作技巧库：风格记忆
└── referenceNotes/        ← 写作技巧库：参考笔记
```
